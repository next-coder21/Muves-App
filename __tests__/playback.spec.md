# Muves Mobile — Playback & Offline Pre-Beta Test Checklist

> Generated by pre-beta audit 2026-05-18.
> Each scenario lists: **Expected Behaviour**, **Pass/Fail Criteria**, and **Code Risk** notes.

---

## 1. Audio Playback — PlayerContext.tsx

### 1.1 Play a song
- **Expected**: Song loads, `isLoading` true then false, audio starts, lock screen shows title/artist.
- **Pass**: `isPlaying === true`, `elapsed` increments, no error banner.
- **Fail**: `isLoading` stays true, `error` is non-null, audio silent.
- **Code Risk**: `manualLoadingRef` cleared only on `status.playing` or `replaceCalledRef && status.isLoaded`; if ExoPlayer never emits `isLoaded=true` (bad URI) `isLoading` stays stuck. Mitigated by the `didJustFinish` cleanup.

### 1.2 Pause / Resume
- **Expected**: `togglePlay` pauses playing audio; second tap resumes from same position.
- **Pass**: `isPlaying` toggles, `elapsed` freezes on pause and continues on resume.
- **Fail**: Tap produces no change, or audio restarts from 0.
- **Code Risk**: `togglePlay` checks `!p.isLoaded || error !== null` and calls `_loadAndPlay` instead of `play` — correct for broken-stream retry. Verify this does not trigger a reload on a healthy paused stream.

### 1.3 Seek (progress bar, ratio 0–1)
- **Expected**: Drag to 50% → audio jumps to halfway point.
- **Pass**: `elapsed ≈ duration * ratio` within ±1 s after seek.
- **Fail**: Seek ignored, no movement, or seek causes crash.
- **Code Risk**: `seek` falls back to `duration` state when `p.duration === 0`; if state `duration` is also 0 the seek is silently skipped. Reproducible on very short songs before metadata loads.

### 1.4 Next — mid queue
- **Expected**: Advances to next song in queue.
- **Pass**: `currentSong` changes, new song starts, previous is pushed to history.
- **Fail**: Same song restarts, or nothing happens.
- **Code Risk**: None — straightforward index increment.

### 1.5 Next — at end of queue, repeat OFF, shuffle OFF
- **Expected**: Playback stops; player stays at last song position; no crash.
- **Pass**: `isPlaying === false`, `currentSong` unchanged (last song), no error.
- **Fail**: Wraps to first song (was bug before fix), crashes, or throws index OOB.
- **Code Risk**: **FIXED** — `next()` previously used `% q.length` (silent wrap). Now checks `nextIdx >= q.length` and pauses. Verify fix is present.

### 1.6 Next — at end of queue, shuffle ON
- **Expected**: Random song (not current) starts.
- **Pass**: New song is different from previous, drawn from queue.
- **Fail**: Same song plays, or OOB access.
- **Code Risk**: `pickRandomIndex` bumps by +1 if `idx === exclude` — this is a weak bias fix (favours index+1), not true Fisher-Yates. Acceptable for UX but document if strict fairness needed.

### 1.7 Prev — within 3 s of song start
- **Expected**: Goes to previous song in queue.
- **Pass**: `currentSong` changes to prior queue entry.
- **Fail**: Restarts current song instead.
- **Code Risk**: Uses `elapsedRef.current > 3` threshold — ensure the ref is always current. Synced via `useEffect` on `elapsed` state, so there is a 1-render lag. Edge case: press prev at exactly 3.0 s may not restart as expected.

### 1.8 Prev — after 3 s
- **Expected**: Restarts current song from 0.
- **Pass**: `elapsed` drops to ~0, same song keeps playing.
- **Fail**: Goes to previous song instead of restarting.
- **Code Risk**: `playerRef.current?.seekTo(0)` — no guard on manualLoadingRef, so a seek-to-0 fires even if another song is being loaded simultaneously. Low risk in practice.

### 1.9 Shuffle toggle
- **Expected**: `isShuffle` flips; next auto-advance picks a random index.
- **Pass**: Repeated `next()` calls produce varied indices, not sequential.
- **Fail**: Sequential order continues after shuffle enabled.
- **Code Risk**: `isShuffleRef` synced via `useEffect` — 1-render lag, acceptable.

### 1.10 Repeat (single-song)
- **Expected**: Song finishes → `seekTo(0)` then `play()` → same song loops.
- **Pass**: `currentSong` unchanged after auto-advance, audio restarts.
- **Fail**: Next song plays instead, or `isLoading` gets stuck.
- **Code Risk**: `seekTo(0).then(() => player.play())` guard checks `manualLoadingRef.current` before calling `play()` — correct. If user presses next while seek is in flight, `manualLoadingRef` is true and the repeat-play is aborted.

### 1.11 Background playback
- **Expected**: Lock screen/notification visible when app is backgrounded; audio continues.
- **Pass**: Audio audible after app backgrounded; lock screen controls visible.
- **Fail**: Audio pauses when app backgrounds; no lock screen controls.
- **Code Risk**: `setAudioModeAsync({ playsInSilentMode: true, shouldPlayInBackground: true })` called on mount — verify this runs before any `createAudioPlayer` call (it does, same useEffect, sequential). iOS silent switch must be OFF for audio to play regardless.

### 1.12 Lock screen controls
- **Expected**: Play/pause/next/prev buttons on lock screen work.
- **Pass**: Tapping lock screen controls reflects in-app `isPlaying`/`currentSong`.
- **Fail**: Controls visible but unresponsive, or metadata wrong (ID3 tags override custom title).
- **Code Risk**: Three-timer retry (200 ms / 800 ms / 2000 ms) in status listener fights ExoPlayer ID3 reads. Verify timers are cancelled before scheduling new ones — they are (`metaTimerRef.current.forEach(clearTimeout)` before `= [...map]`). Rapid song skipping stress test recommended.

### 1.13 didJustFinish — no double-advance
- **Expected**: Each song finish triggers exactly one auto-advance.
- **Pass**: One new song starts; queue index increments by exactly 1.
- **Fail**: Two songs are skipped, or the same `_loadAndPlay` is called twice.
- **Code Risk**: `didJustFinish` fires inside `playbackStatusUpdate`. If the status listener fires twice with `didJustFinish=true` for the same finish event, `_loadAndPlay` would be called twice. Mitigated by `loadIdRef` — the second call with stale `myId` is a no-op. Verify with verbose logging in QA build.

### 1.14 Network error during stream
- **Expected**: `error` state is populated; error message shown to user; no crash; tap play retries.
- **Pass**: Error banner/text visible; tap play triggers `_loadAndPlay` (detected by `!p.isLoaded || error !== null` check in `togglePlay`).
- **Fail**: App crashes; infinite spinner; no retry mechanism.
- **Code Risk**: `_loadAndPlay` catch block sets `error` — but `player.replace()` errors may surface asynchronously (not synchronously). The async error path (stream drops mid-play) only sets `isBuffering`; `error` is never set for mid-stream drop. **Gap**: user gets buffering spinner with no message, never an error state.

## 2. ExoPlayer Metadata — patches/expo-audio+1.1.1.patch

### 2.1 overrideExoPlayerMetadata replaces correct MediaItem
- **Expected**: Lock screen shows correct title/artist, not ID3 from audio file.
- **Pass**: Title/artist match app metadata, not file tags.
- **Fail**: Wrong title shown, or old song metadata persists after track change.
- **Code Risk**: `val index = ref.currentMediaItemIndex` — snapshot taken at call time. If ExoPlayer has already advanced the media queue internally by the time the call executes, the wrong index is replaced. Low risk because `replaceMediaItem` is called immediately after `setActiveForLockScreen`.

### 2.2 Three-timer retry pattern stored and cancelled
- **Expected**: On song change, previous retries do not overwrite new song's metadata.
- **Pass**: After rapid skip ×3, final song's title/artist on lock screen.
- **Fail**: Lock screen reverts to a previous song's title after 200 ms / 800 ms / 2000 ms.
- **Code Risk**: `metaTimerRef.current.forEach(clearTimeout)` runs before new timers are assigned — correct. The `snapId !== loadIdRef.current` guard inside each timer provides double safety.

### 2.3 Local song cover art not fetched from server
- **Expected**: Local songs show a generic placeholder, not a failed server request.
- **Pass**: No 404 network requests for local song cover art; placeholder shown.
- **Fail**: Network error in console `GET /cover/1234 404`, or broken image.
- **Code Risk**: **FIXED** — `buildMeta` previously used `song._id` as the cover URL gate, but local songs carry MediaLibrary IDs as `_id` which are not valid server cover IDs. Fix: gate on `!song.localPath && song._id`.

### 2.4 Artwork URI not set on ExoPlayer MediaItem
- **Expected**: Album art appears on Android lock screen / notification.
- **Pass**: Artwork visible in notification shade.
- **Fail**: Blank/placeholder art in notification despite artwork URI being available.
- **Code Risk**: `overrideExoPlayerMetadata` builds `exoMeta` without `.setArtworkUri()`. Artwork is delivered separately via `AudioControlsService.updateMetadata`. Verify the service path sets artwork; if not, add `.setArtworkUri(URI.parse(metadata.artworkUrl))` to the patch.

## 3. Offline / Local Songs

### 3.1 Permission granted → loadSongs() called immediately
- **Expected**: After `requestPermission()` returns `true`, `localSongs` is populated synchronously before next render.
- **Pass**: `localSongs.length > 0` immediately after permission grant (same async call chain).
- **Fail**: `localSongs` is empty until user navigates away and back.
- **Code Risk**: `requestPermission` calls `await loadSongs()` before returning — correct, no waiting for next render.

### 3.2 Pagination — more than 200 audio files
- **Expected**: All files discovered, not just first 200.
- **Pass**: `localSongs.length` equals actual file count even with 201+ files.
- **Fail**: Exactly 200 songs shown regardless of actual count.
- **Code Risk**: `fetchAllAudio` uses a `while(true)` loop with `after = page.endCursor` and `if (!page.hasNextPage) break` — correct pagination. Verify `endCursor` is valid across Android/iOS versions.

### 3.3 Local URI — no Authorization header sent
- **Expected**: Local file requests do not include Bearer token.
- **Pass**: No `Authorization` header in network log for `file://` URIs.
- **Fail**: Token sent on local requests (privacy/security issue, no functional impact).
- **Code Risk**: `_loadAndPlay` gates header on `!localPath && tokenRef.current` — correct. `localPath` is `song.localPath ?? resolveLocalPath(song._id)`, so header is suppressed for both downloaded and MediaLibrary songs.

### 3.4 resolveLocalPath returns correct URI
- **Expected**: `resolveLocalPath(id)` returns the `file://` URI for a song that has been indexed.
- **Pass**: URI returned matches `asset.uri` from MediaLibrary.
- **Fail**: `undefined` returned for a song that is locally available.
- **Code Risk**: `_localPathResolver` is a module-level reference set in `LocalSongsProvider`'s `useEffect`. If `PlayerContext` calls `resolveLocalPath` before `LocalSongsProvider` mounts, it returns `undefined`. Render order in `_layout.tsx` must wrap `LocalSongsProvider` outside `PlayerProvider`.

## 4. Network Transitions

### 4.1 WiFi → Mobile data: playback continues
- **Expected**: Switching network type does not interrupt audio; `isOnline` stays `true`.
- **Pass**: Audio uninterrupted during switch; no error banner; `isOnline` remains `true`.
- **Fail**: Audio drops, error shown, or `isOnline` briefly `false`.
- **Code Risk**: `NetInfo.addEventListener` fires on type change. `isInternetReachable` may briefly be `null` (unknown) during switch, causing `online = false` momentarily. `isInternetReachable !== false` in the condition correctly treats `null` as online.

### 4.2 Online → Offline: banner shows, no crash
- **Expected**: `isOnline` becomes `false`; offline banner visible; playing audio continues (already buffered).
- **Pass**: Banner rendered with local song count; no JS exception; in-progress audio continues.
- **Fail**: App crashes; no banner; audio immediately stops.
- **Code Risk**: Home screen `trending` swaps to `localSongs` reactively when `isOnline` is `false` — correct reactive render. Audio player is unaffected by `isOnline` — it's managed independently.

### 4.3 Offline → Online: home screen reloads content
- **Expected**: When device reconnects, home screen fetches fresh songs/artists automatically.
- **Pass**: New songs appear without manual pull-to-refresh.
- **Fail**: Home screen stays empty/stale until user manually refreshes.
- **Code Risk**: **FIXED** — Home screen previously only loaded on mount. Added `useEffect` watching `isOnline` that calls `load()` when transitioning from `false` → `true` (using `prevOnlineRef` to detect rising edge only, preventing reload on initial mount).

### 4.4 Offline mode manual toggle
- **Expected**: `setOfflineMode(true)` persists; reverts to `false` automatically when back online.
- **Pass**: `isOfflineMode === false` after reconnect (via `if (online) setIsOfflineMode(false)` in listener).
- **Fail**: `isOfflineMode` stays `true` after reconnect.
- **Code Risk**: Auto-reset on reconnect is implemented in `NetworkContext` — verify the offline mode UI toggle (if any) re-reads `isOfflineMode` from context reactively.
